# ITEM71) 필요 없는 검사 예외 사용은 피하라

### 1. 검사 예외의 장단점

- API와 프로그램의 질을 높일 수 있다.
- API 사용자에게 부담을 준다.
- 검사 예외를 던지는 메서드는 스트림 안에서 직접 사용할 수 없다.

### 2. 검사 예외와 비검사 예외 판단 기준

- API를 제대로 사용해도 발생할 수 있는 예외이거나, 프로그래머가 의미 있는 조치를 취할 수 있는 경우가 아니면 비검사 예외 사용이 좋다.
- 단 하나의 검사 예외만 던질 경우
    - API 사용자는 try 블록을 추가해야 하고, 스트림에서 직접 사용하지 못한다.
        
        → 이런 경우 검사 예외를 안 던지는 방법 고민해봐야 함.
        

### 3. 검사 예외를 회피하는 방법

- Optional 반환
    - 가장 쉬운 방법
    - 예외가 발생한 이유를 알려주는 부가 정보를 담을 수 없다.
- 검사 예외를 던지는 메서드를 2개로 쪼개 비검사 예외로 바꾸기
    - 첫 번째 메서드는 예외가 던져질지 여부를 boolean으로 반환한다.
    
    ```java
    // 리펙토링 전
    try {
    	obj.action(args);
    } catch (TheCheckedException e) {
    	...// 예외 상황에 대처한다.
    }
    // 리펙토링 후
    if(obj.actionRermitted(args)) {
     obj.action(args);
    } else {
    	... // 예외 상에 대처한다.
    }
    // 프로그래머가 이 메서드가 성공하리라는 걸 안다거나 실패 시 스레드 중단을 원하면
    obj.action(args);
    ```
